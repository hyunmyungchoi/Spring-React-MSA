# 채팅 내구성·확장 계획

- 작성일: 2026-07-17
- 대상: Member BFF 채팅 모듈, PostgreSQL, Redis, Kafka
- 관련 결정: ADR-003

## 현재 기준선

- 인증 session 기반 raw WebSocket
- PostgreSQL room/message 저장
- 최근 메시지 Redis cache
- Redis pub/sub 기반 replica fan-out
- Kafka created event, notification/analytics groups, retry와 DLT
- REST history와 2-replica load test script

핵심 미완료 항목은 Transactional Outbox와 consumer 멱등성이다.

## 1단계: 데이터·동시성 안전성

- 승인한 Flyway SQL Migration으로 `member_bff` Schema의 Chat Table Version을 관리한다.
- 여러 replicas가 같은 room을 최초 생성할 때 unique 충돌을 재조회로 복구한다.
- message content, room, sender field DB 길이 제약을 Java validation과 맞춘다.
- history pagination cursor를 추가해 최근 200개 제한 너머를 조회할 수 있게 한다.
- 데이터 보존·삭제·개인정보 정책을 정한다.

완료 조건:

- 두 replicas의 동시 첫 메시지 저장이 실패하지 않는다.
- migration을 빈 DB와 기존 DB 모두에 적용할 수 있다.

## 2단계: Transactional Outbox

권장 schema:

- `event_id` unique
- `aggregate_type`, `aggregate_id`, `event_type`
- `partition_key`, `payload`, `occurred_at`
- `status`, `attempts`, `next_attempt_at`, `published_at`, `last_error`

relay는 짧은 batch와 `FOR UPDATE SKIP LOCKED` 또는 동등한 claim 전략을 사용한다. Kafka ack 후 published로 표시하고, 지수 backoff와 최대 시도 정책을 둔다. 영구 실패는 운영자가 재처리할 수 있어야 한다.

완료 조건:

- DB commit 후 프로세스를 강제 종료해도 재기동 시 event가 발행된다.
- Kafka 장시간 장애 중 message write는 성공하고 outbox가 적체된다.
- 복구 후 순차적으로 drain된다.

## 3단계: Consumer 멱등성과 실제 기능

- `eventId` 처리 기록 또는 idempotent sink를 구현한다.
- notification/analytics consumer를 단순 로그에서 실제 use case로 분리한다.
- DLT re-drive 절차와 권한을 정의한다.
- schema compatibility test를 공통 Kafka contract 모듈에 추가한다.
- payload에 schema version을 추가한다.

## 4단계: WebSocket 복원력

- client reconnect backoff/jitter와 최대 재시도
- server ping/pong idle timeout
- connection/session/room별 제한과 최대 frame 크기
- slow consumer send queue 제한
- rolling update drain과 graceful close
- Redis pub/sub 실패·복구 시 gap reconciliation

## 5단계: 관측성

필수 metric:

- active connections, connect failures
- messages in/out, validation errors
- persist latency, broadcast latency
- Redis publish/subscriber errors
- outbox pending/oldest age/retry/failure
- Kafka publish latency, consumer lag, DLT count

trace에는 WebSocket connection ID 대신 안전한 fingerprint, room ID, message/event ID를 넣고 content는 기록하지 않는다.

## 부하·장애 시험

- 2 replicas에서 100/500/1,000 connections 단계 시험
- 고정 message rate와 burst
- Redis 중단, Kafka 중단, PostgreSQL 지연, pod rolling restart
- 수신 누락, 중복, 순서, p50/p95/p99 측정
- reconnect 후 history로 gap 복구 검증

## 분리 판단

connection 수와 배포 주기가 BFF HTTP workload를 지배하면 Chat Service로 분리한다. 분리 시 BFF session을 검증하는 handshake token 또는 internal session introspection 계약이 추가로 필요하다.
